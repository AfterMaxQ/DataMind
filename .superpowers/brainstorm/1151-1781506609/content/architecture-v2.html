DataMind — Technical Architecture

Six engineering innovations that solve the AI context problem for data science

Innovation 1: Event-Sourced Knowledge Graph

Every AI action is an immutable event. The graph is a materialized view. Replay from any point.

                    ┌────────────────────────────────┐
                    │       KNOWLEDGE GRAPH           │
                    │       (SQLite + graph layer)    │
                    └────────────────────────────────┘
                                     ▲
                         materialized view
                                     │
    ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ Dataset  │ │  Script  │ │Execution │ │Decision  │
    │  Node    │ │  Node    │ │  Node    │ │  Node    │
    └──────────┘ └──────────┘ └──────────┘ └──────────┘
          ▲            ▲            ▲            ▲
          │            │            │            │
    ┌─────┴────────────┴────────────┴────────────┴─────┐
    │              IMMUTABLE EVENT LOG                  │
    │                                                  │
    │  executions/2026-06-15-001-clean-nulls.md        │
    │  executions/2026-06-15-002-explore-revenue.md     │
    │  executions/2026-06-15-003-build-model.md         │
    └──────────────────────────────────────────────────┘

Innovation 2: The Project Structure

Not a folder convention — it's a schema that the AI natively understands

DataMind Project/
├── .datamind/
│   ├── graph.db              ← knowledge graph (SQLite)
│   ├── context/              ← auto-generated context files
│   │   ├── PROJECT.md        ← project overview
│   │   ├── DATASETS.md       ← all schemas + statistics
│   │   ├── HISTORY.md        ← recent activity
│   │   └── CHECKPOINT.md     ← compressed understanding
│   └── config.yaml
├── data/
│   ├── raw/          ← immutable (never modified)
│   └── processed/    ← all derived datasets
├── scripts/          ← AI-generated, linked to lineage
├── describe/         ← auto data.describe for every dataset
└── executions/       ← immutable per-session logs

Innovation 3: Context Packing Algorithm

Not RAG. Not vector search. Priority-ordered structured injection.

┌─────────────────────────────────────────────────┐
│            CONTEXT PACKER                        │
│                                                  │
│  Input: user query + project state               │
│                                                  │
│  Priority 1 (ALWAYS):                            │
│    • PROJECT.md       ~500 tokens                │
│    • All DATASETS.md  ~200 tokens each            │
│                                                  │
│  Priority 2 (RECENT):                            │
│    • Last 5 execution logs   ~2k tokens each     │
│    • Last 10 decisions       ~100 tokens each    │
│                                                  │
│  Priority 3 (RELEVANT):                          │
│    • Graph search for query-related nodes        │
│    • e.g. "revenue" → find datasets/scripts      │
│              with "revenue" in metadata          │
│                                                  │
│  Priority 4 (COMPRESSED):                        │
│    • CHECKPOINT.md — dense summary of            │
│      all prior work, periodically regenerated    │
│                                                  │
│  Output: ~30-50k token context pack              │
└─────────────────────────────────────────────────┘

Innovation 4: Script-as-Edge Pattern

The script IS the relationship. No external mapping. Read the graph = understand the data.

     raw/sales.csv ──→ scripts/clean_nulls.py ──→ processed/sales_clean.csv
            │                                          │
            │              ┌───────────────────────────┘
            │              ▼
            │    scripts/aggregate.py ──→ processed/sales_agg.parquet
            │                                   │
            └──────→ scripts/merge.py ──────────┘
                            │
                            ▼
                  processed/master_table.csv

    Each edge = the script that transformed the data.
    AI reads scripts/ dir → instantly knows all transformations.

Innovation 5: Auto-Describe Engine

Every dataset gets a machine-readable description the AI never needs to rediscover

describe/sales_2024.describe.md:

  ## sales_2024.csv
  - Rows: 45,231 | Columns: 12 | Size: 3.2 MB
  - Ingested: 2026-06-15 14:32

  | Column      | Type    | Null%  | Unique | Range/Values          |
  |-------------|---------|--------|--------|------------------------|
  | date        | date    | 0%     | 365    | 2024-01-01..2024-12-31|
  | region      | str     | 2.1%   | 7      | N. America, EU, APAC..|
  | revenue     | float   | 23.4%  | -      | -$50..$49,999 (1 neg) |
  | product_id  | str     | 0%     | 1,247  | SKU-0001..SKU-1247    |

  AI never re-reads the CSV to understand its structure.

Innovation 6: Checkpoint — Compressed Understanding

Like git packfiles for knowledge. Periodically regenerate a dense state-of-project snapshot.

How This Solves the Core Problem

New AI session starts → AI immediately knows:

  • Every dataset's schema, stats, and meaning (DATASETS.md)
  • How each dataset was created (lineage graph → scripts)
  • What was tried, what worked, what failed (execution logs)
  • Why decisions were made (decision nodes)
  • The current state of the project (CHECKPOINT.md)
  • What to do next (suggested next steps from checkpoint)

Open challenges to resolve:

  • Checkpoint regeneration: when and how often?
  • How to handle manual (non-AI) changes to data?
  • Multi-user collaboration: merge conflict in graph.db?
  • Large datasets: describe generation cost vs caching
